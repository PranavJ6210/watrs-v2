# WATRS v2.0 — Frontend Development Roadmap (Phase 2)

> **Focus:** Community Ingestion, Admin Moderation, and Gamification
> **Stack Integration:** React Router v7, Context API/Zustand, Tailwind CSS

---

## 1. Project Alignment & Objectives

The current frontend prototype successfully demonstrates the search and recommendation UI. However, to fulfill the complete WATRS v2.0 specification, the application must transition from a read-only search tool into a **community-driven platform**. 

The next phase of frontend development focuses on building the interfaces required for the **Data Ingestion Workflow (Staging)**, **Role-Based Access Control (RBAC)**, and the **Gamification Feedback Loop**.

---

## 2. Phase 1: Authentication & State Management

Before users can contribute or admins can moderate, the app requires a robust authentication layer to handle the JWTs generated by the FastAPI backend.

* **Global Auth State:** Implement a state manager (e.g., Zustand or React Context) to store the current user session, JWT token (if not using HttpOnly cookies), and the user's role (`Explorer`, `Contributor`, `Pathfinder`, `Super-Admin`).
* **Protected Routes:** Create wrapper components in React Router v7 to protect specific paths:
    * `/contribute` $\rightarrow$ Requires `Contributor` or higher.
    * `/admin` $\rightarrow$ Requires `Super-Admin`.
* **Auth UI:** Develop login and registration modals/pages with secure form handling and error states.

---

## 3. Phase 2: The Contributor Workflow (Data Intake)

This is the interface where users submit new "Hidden Gems" to the `places_staging` collection. To maintain data quality, this form must be strictly controlled.

### 3.1 The Multi-Step Submission Form
Build a cohesive, multi-step form at `/contribute`:

* **Step 1: Location Definition**
    * Integrate a Google Places Autocomplete search bar so the user can select the place.
    * Automatically extract the `google_place_id`, `name`, `lat`, and `lon` to ensure backend data cleanliness.
* **Step 2: The "Vibe" Taxonomy (Strict Enums)**
    * Render a multi-select UI for `watrs_tags`.
    * **Crucial:** Do not allow free-text entry. Map this to the predefined backend categories (e.g., *Waterfall*, *Trekking*, *Heritage*).
* **Step 3: Safety & Accessibility Checklist**
    * Create radio buttons or dropdowns for `road_access` (Paved, Unpaved, 4WD Only).
    * Add toggles for `family_safe` and `cell_service`.
* **Step 4: Media Upload**
    * Implement an image uploader that interfaces with Cloudinary (or passes the file to the backend to handle the Cloudinary upload).

---

## 4. Phase 3: The Admin & Pathfinder Portal

Data sitting in `places_staging` needs a UI for review before it hits the live recommendation engine.

### 4.1 The Moderation Dashboard (`/admin`)
* **Staging Queue:** Develop a table or Kanban-style board fetching data from a new `/api/v1/staging` endpoint.
* **Review Interface:** Clicking a staging item opens a detailed view showing the submitted location, tags, and safety metadata.
* **Action Buttons:**
    * **Approve:** Moves document to `places_live`.
    * **Edit & Approve:** Allows the Admin to correct tags or adjust safety ratings before approving.
    * **Reject:** Deletes the staging document (with an optional text input for the rejection reason).

---

## 5. Phase 4: Gamification & The User Profile

To solve the "Cold Start" and data quality problems, the frontend must incentivize users to submit and verify locations.

### 5.1 User Dashboard (`/profile`)
* **Trust Score & Metrics:** Display the user's total approved submissions and community feedback score.
* **The "Pathfinder" Tracker:** Implement a visual progress bar showing how close the user is to earning the "Pathfinder" badge (e.g., "3 more verified submissions to unlock Pathfinder status!").

### 5.2 Enhancing the PlaceCard
* Update the existing `PlaceCard.jsx` to show the submitter's username or a "Verified by Pathfinder" badge to build community trust.

---

## 6. Phase 5: Refining the Core Search UI

The prototype search page needs functional wiring to leverage the backend's full capabilities.

* **Dynamic Radius Slider:** Replace the hardcoded 500km radius with a user-adjustable slider (e.g., 10km to 200km).
* **Tag Filters:** Add a horizontal scrolling row of "Vibe" chips at the top of the `/search` page so users can explicitly pass `tags` to the backend query.
* **Base Location Override:** Allow the user to manually type a base city (e.g., "Madurai") instead of relying purely on browser geolocation, accommodating users planning future trips.

---
